# Authorization

NestJS CRUD Automator ships with a single authorization pipeline and two first-class authoring modes:

- `hooks`: entity-scoped `@ApiAuthorizationPolicy({ entity })` classes
- `iam`: storage-agnostic policy attachments and IAM documents

`@ApiControllerSecurable()` is marker-only. It enables the authorization pipeline for a controller, but the selected mode and all authorization configuration live in `@ApiController({ authorization: ... })`. A route can override the controller default with `routes[routeType].authorization.mode`, but each route still runs in exactly one mode.

## Module Setup

Import `ApiAuthorizationModule` once. Use `forRoot()` for plain value-based setup. Use `forRootAsync()` when `principalResolver`, `hookPermissionSources`, `iam.attachmentSources`, or `iam.documentSources` must come from Nest DI.

```typescript filename="app.module.ts" copy
import type {
	IApiAuthorizationPrincipal,
	IApiHookPermissionSource,
	IApiPolicyAttachmentSource,
	IApiPolicyDocumentSource,
	IApiResolvedPolicyAttachments,
} from "@elsikora/nestjs-crud-automator";

import { Module } from "@nestjs/common";
import { TypeOrmModule } from "@nestjs/typeorm";

import {
	ApiAuthorizationModule,
	EApiAuthorizationPrincipalType,
	EApiPolicySourceType,
	AuthorizationResolveDefaultPrincipal,
} from "@elsikora/nestjs-crud-automator";

const hookPermissionSource: IApiHookPermissionSource = {
	async getPermissions(principal: IApiAuthorizationPrincipal): Promise<ReadonlyArray<string>> {
		const permissions = principal.claims?.permissions;

		return Array.isArray(permissions)
			? permissions.filter((value): value is string => typeof value === "string")
			: [];
	},
};

const iamAttachmentSource: IApiPolicyAttachmentSource = {
	async getAttachments(principal): Promise<IApiResolvedPolicyAttachments> {
		return {
			attachments: [
				{
					policyId: "user-items",
					principalId: principal.id,
					principalType: principal.type,
				},
			],
			boundaries: [],
		};
	},
};

const iamDocumentSource: IApiPolicyDocumentSource = {
	async getDocumentsByIds(ids) {
		return ids.map((id) => ({
			document: {
				Statement: [
					{
						Action: ["admin:item:list", "admin:item:read"],
						Condition: {
							StringEquals: {
								"resource.operatorId": "operator-1",
							},
						},
						Effect: "Allow",
						Resource: ["gameport:admin:item/{id}"],
						Sid: "AllowOperatorItems",
					},
				],
				Version: "2012-10-17",
			},
			id,
			namespace: "admin:item",
			sourceType: EApiPolicySourceType.MANAGED,
			version: "2026-03-14",
		}));
	},
};

@Module({
	imports: [
		TypeOrmModule.forRoot(/* ... */),
		ApiAuthorizationModule.forRoot({
			hookPermissionSources: [hookPermissionSource],
			iam: {
				attachmentSources: [iamAttachmentSource],
				documentSources: [iamDocumentSource],
			},
			principalResolver: {
				resolve(user) {
					if (!user || typeof user !== "object" || !("account" in user)) {
						return AuthorizationResolveDefaultPrincipal(user);
					}

					const payload = user as {
						account: { id: string; operatorId: string };
						access: { permissions: Array<string>; roles: Array<string> };
					};

					return {
						attributes: { operatorId: payload.account.operatorId },
						claims: { permissions: payload.access.permissions },
						id: payload.account.id,
						roles: payload.access.roles,
						type: EApiAuthorizationPrincipalType.USER,
					};
				},
			},
		}),
	],
})
export class AppModule {}
```

The built-in `AuthorizationResolveDefaultPrincipal()` maps `request.user` into an `IApiAuthorizationPrincipal` using `id | uuid | email`, `roles | role`, `groups | group`, and `claims`. The principal intentionally does not contain permissions or policy attachments:

- hooks mode resolves permissions through `hookPermissionSources`
- IAM mode resolves attachments and documents through `iam.attachmentSources` and `iam.documentSources`

### `forRootAsync()` for DI-backed sources

Use `forRootAsync()` when your resolver or IAM sources depend on `Repository`, `DataSource`, or other Nest providers. The authorization module supports `imports`, `inject`, `useFactory`, `useClass`, and `useExisting`. The public API also exports `IApiAuthorizationModuleAsyncOptions` and `IApiAuthorizationModuleOptionsFactory` for typed Nest module wiring.

```typescript filename="authorization/authorization.module.ts" copy
import type {
	IApiAuthorizationPrincipal,
	IApiAuthorizationPrincipalResolver,
	IApiPolicyAttachmentSource,
	IApiPolicyDocumentRecord,
	IApiPolicyDocumentSource,
	IApiResolvedPolicyAttachments,
} from "@elsikora/nestjs-crud-automator";

import { Injectable, Module } from "@nestjs/common";
import { InjectRepository, TypeOrmModule } from "@nestjs/typeorm";
import { Repository } from "typeorm";

import {
	ApiAuthorizationModule,
	AuthorizationResolveDefaultPrincipal,
	EApiAuthorizationPrincipalType,
	EApiPolicySourceType,
} from "@elsikora/nestjs-crud-automator";

@Injectable()
class DbPrincipalResolver implements IApiAuthorizationPrincipalResolver {
	public resolve(user: unknown): IApiAuthorizationPrincipal {
		return AuthorizationResolveDefaultPrincipal(user);
	}
}

@Injectable()
class DbAttachmentSource implements IApiPolicyAttachmentSource {
	public constructor(
		@InjectRepository(PolicyAttachmentEntity)
		private readonly repository: Repository<PolicyAttachmentEntity>,
	) {}

	public async getAttachments(principal: IApiAuthorizationPrincipal): Promise<IApiResolvedPolicyAttachments> {
		const rows = await this.repository.find({
			where: { principalId: principal.id },
		});

		return {
			attachments: rows.map((row) => ({
				policyId: row.policyId,
				principalId: row.principalId,
				principalType: EApiAuthorizationPrincipalType.USER,
			})),
			boundaries: [],
		};
	}
}

@Injectable()
class DbDocumentSource implements IApiPolicyDocumentSource {
	public constructor(
		@InjectRepository(PolicyDocumentEntity)
		private readonly repository: Repository<PolicyDocumentEntity>,
	) {}

	public async getDocumentsByIds(ids: ReadonlyArray<string>): Promise<ReadonlyArray<IApiPolicyDocumentRecord>> {
		const rows = await this.repository.find();

		return rows
			.filter((row) => ids.includes(row.id))
			.map((row) => ({
				document: row.document,
				id: row.id,
				namespace: row.namespace,
				sourceType: EApiPolicySourceType.DATABASE,
				version: row.version,
			}));
	}
}

@Module({
	exports: [DbAttachmentSource, DbDocumentSource, DbPrincipalResolver],
	imports: [TypeOrmModule.forFeature([PolicyAttachmentEntity, PolicyDocumentEntity])],
	providers: [DbAttachmentSource, DbDocumentSource, DbPrincipalResolver],
})
class AuthorizationSourcesModule {}

@Module({
	imports: [
		AuthorizationSourcesModule,
		ApiAuthorizationModule.forRootAsync({
			imports: [AuthorizationSourcesModule],
			inject: [DbAttachmentSource, DbDocumentSource, DbPrincipalResolver],
			useFactory: (attachmentSource: DbAttachmentSource, documentSource: DbDocumentSource, principalResolver: DbPrincipalResolver) => ({
				iam: {
					attachmentSources: [attachmentSource],
					documentSources: [documentSource],
				},
				principalResolver,
			}),
		}),
	],
})
export class AppModule {}
```

If you prefer Nest factory classes, the same setup can be expressed with `useClass` or `useExisting`.

## Controller Opt-In and Mode Selection

Use both decorators together on any secured controller:

```typescript filename="controllers/user.controller.ts" copy
import { EApiRouteType } from "@elsikora/nestjs-crud-automator";

import {
	EApiAuthorizationMode,
	ApiController,
	ApiControllerSecurable,
} from "@elsikora/nestjs-crud-automator";

@ApiControllerSecurable()
@ApiController<UserEntity>({
	authorization: {
		defaultMode: EApiAuthorizationMode.HOOKS,
	},
	entity: UserEntity,
	path: "users",
	routes: {
		[EApiRouteType.GET_LIST]: {
			authorization: {
				mode: EApiAuthorizationMode.IAM,
			},
		},
	},
})
export class UserController {
	constructor(public service: UserService) {}
}
```

Bootstrap validation fails fast when:

- `@ApiControllerSecurable()` is present but the controller has no `authorization` block
- a controller declares `authorization` but is missing `@ApiControllerSecurable()`
- the selected mode is not `hooks` or `iam`

## Runtime authorization actions

`@ApiMethod(...)` uses two separate action concepts:

- `action` is a documentation hint used for generated Swagger summaries and descriptions
- `authorization.action` is the runtime authorization action string used by hooks and IAM

Auto-generated CRUD routes receive built-in runtime actions automatically:

- `create`
- `delete`
- `get`
- `getList`
- `partialUpdate`
- `update`

Custom secured routes should declare their own domain action strings:

```typescript filename="controllers/user.controller.ts" copy
@ApiMethod<UserEntity>({
	action: EApiAction.UPDATE,
	authorization: {
		action: "update.promote",
	},
	entity: UserEntity,
	httpCode: HttpStatus.OK,
	method: RequestMethod.POST,
	path: ":id/promote",
	responseType: UserResponseDto,
})
public promote(@Param("id") id: string) {
	return this.service.promote(id);
}
```

That same `authorization.action` value is what hooks receive as `context.action` and what IAM turns into a namespaced action such as `admin:user:update.promote`.

## Hooks Mode

Hooks mode keeps the original policy authoring style, but now runs on `principal + permissions` rather than `subject + subject.permissions`. Policies are still auto-discovered by entity through `@ApiAuthorizationPolicy({ entity })`.

```typescript filename="policies/user-hooks.policy.ts" copy
import type {
	IApiAuthorizationRuleContext,
	IApiAuthorizationScope,
	TApiAuthorizationPolicyBeforeGetListContext,
	TApiAuthorizationPolicyBeforePartialUpdateContext,
} from "@elsikora/nestjs-crud-automator";

import {
	EApiAuthorizationPermissionMatch,
	ApiAuthorizationPolicy,
	ApiAuthorizationPolicyBase,
} from "@elsikora/nestjs-crud-automator";

@ApiAuthorizationPolicy<UserEntity>({
	entity: UserEntity,
	priority: 200,
})
export class UserHooksPolicy extends ApiAuthorizationPolicyBase<UserEntity> {
	private getOperatorId(principal: IApiAuthorizationRuleContext<UserEntity>["principal"]): string | undefined {
		return principal.attributes.operatorId as string | undefined;
	}

	private scopeToOperator(context: IApiAuthorizationRuleContext<UserEntity>): IApiAuthorizationScope<UserEntity> {
		return {
			where: {
				operatorId: this.getOperatorId(context.principal),
			},
		};
	}

	public onBeforeGetList(context: TApiAuthorizationPolicyBeforeGetListContext<UserEntity>) {
		if (context.query.filters?.operatorId && context.query.filters.operatorId !== this.getOperatorId(context.principal)) {
			return [];
		}

		return [
			...this.allowForRoles(["platform-admin"]),
			...this.allowForPermissions(["admin.user.read", "admin.user.list"], {
				match: EApiAuthorizationPermissionMatch.ALL,
				scope: (ruleContext: IApiAuthorizationRuleContext<UserEntity>): IApiAuthorizationScope<UserEntity> => this.scopeToOperator(ruleContext),
			}),
		];
	}

	public onBeforePartialUpdate(context: TApiAuthorizationPolicyBeforePartialUpdateContext<UserEntity>) {
		if (context.body.operatorId && context.body.operatorId !== this.getOperatorId(context.principal)) {
			return [];
		}

		return [
			...this.allowForPermissions(["admin.user.update"], {
				scope: (ruleContext: IApiAuthorizationRuleContext<UserEntity>): IApiAuthorizationScope<UserEntity> => this.scopeToOperator(ruleContext),
			}),
			...this.denyForPermissions(["admin.user.update"], {
				condition: ({ principal }: IApiAuthorizationRuleContext<UserEntity>) => Boolean(principal.attributes.isOperatorLocked),
				priority: 1000,
			}),
		];
	}
}
```

Generated CRUD routes dispatch to CRUD hooks such as `onBeforeGetList` or `onBeforeUpdate` by internal `routeType`. Custom `@ApiMethod(...)` routes do not use CRUD hook names; handle them in `getCustomActionRule(action, context)` instead:

```typescript filename="policies/user-hooks.policy.ts" copy
public getCustomActionRule(action: string) {
	if (action === "update.promote") {
		return this.allowForPermissions(["admin.user.promote"]);
	}

	return [];
}
```

Hook contexts expose:

- `principal` and resolved `permissions`
- `body`, `parameters`, `query`, `headers`, and `ip`
- `action`, `routeType`, `entity`, and `entityMetadata`
- the same data again under `context.DATA`

`routeType` is only populated for auto-generated CRUD handlers. For custom `@ApiMethod(...)` routes, branch on `action`.

For stronger typing, use the route-aware helper contexts:

- `TApiAuthorizationPolicyBeforeCreateContext<E>`
- `TApiAuthorizationPolicyBeforeGetContext<E>`
- `TApiAuthorizationPolicyBeforeGetListContext<E>`
- `TApiAuthorizationPolicyBeforeUpdateContext<E>`
- `TApiAuthorizationPolicyBeforePartialUpdateContext<E>`

Permission helpers compare required permissions against the permissions returned by your hook permission sources:

- `EApiAuthorizationPermissionMatch.ANY` is the default
- `EApiAuthorizationPermissionMatch.ALL` requires every listed permission
- wildcards are supported on granted permissions only, for example `admin.user.*`, `admin.*`, or `*`

If you need custom checks outside a policy class, use `AuthorizationPermissionMatches()` and `AuthorizationPermissionSetMatches()`.

Policy caching remains available in hooks mode through `ApiAuthorizationPolicyRegistry.configureCache()` or per policy with the `cache` option. Leave caching disabled when a policy depends on request-specific data.

## IAM Mode

IAM mode does not use `@ApiAuthorizationPolicy` classes. Instead, the runtime:

1. Resolves a principal
2. Loads policy attachments for that principal
3. Loads IAM documents by attachment policy IDs
4. Evaluates identity policies and optional boundaries
5. Compiles safe resource scopes into controller queries

Every IAM-enabled controller must declare:

- `defaultMode: EApiAuthorizationMode.IAM`
- `policyNamespace`
- `resourceDefinition`

```typescript filename="controllers/item.controller.ts" copy
import { ApiController, ApiControllerSecurable, EApiAuthorizationMode } from "@elsikora/nestjs-crud-automator";

@ApiControllerSecurable()
@ApiController<ItemEntity>({
	authorization: {
		defaultMode: EApiAuthorizationMode.IAM,
		policyNamespace: "admin:item",
		resourceDefinition: {
			entity: ItemEntity,
			fields: [
				{
					isFilterable: true,
					path: "resource.id",
					queryPath: "id",
				},
				{
					isFilterable: true,
					path: "resource.operatorId",
					queryPath: "operator.id",
				},
				{
					isFilterable: true,
					path: "resource.status",
					queryPath: "status",
				},
			],
			namespace: "admin:item",
			resourcePath: "gameport:admin:item/{id}",
			resourceType: "gameport:admin:item",
		},
	},
	entity: ItemEntity,
	path: "items",
})
export class ItemController {
	constructor(public service: ItemService) {}
}
```

Generated CRUD actions are normalized to IAM-friendly names inside the configured namespace:

- `get` -> `<policyNamespace>:read`
- `getList` -> `<policyNamespace>:list`
- `create` -> `<policyNamespace>:create`
- `update` / `partialUpdate` -> `<policyNamespace>:update`
- `delete` -> `<policyNamespace>:delete`

Custom `@ApiMethod(...)` actions pass through unchanged after the namespace. For example, `authorization.action: "update.promote"` becomes `admin:item:update.promote` when `policyNamespace` is `admin:item`.

`queryPath` may be nested, such as `operator.id`, when your repository filter shape uses relations. The query planner now safely compiles both `resource.id` and relation-backed paths such as `resource.operatorId` for `GET` and `GET_LIST`.

Example IAM sources:

```typescript filename="authorization/iam-sources.ts" copy
import type {
	IApiPolicyAttachmentSource,
	IApiPolicyDocumentSource,
	IApiResolvedPolicyAttachments,
} from "@elsikora/nestjs-crud-automator";

import {
	EApiAuthorizationPrincipalType,
	EApiPolicySourceType,
} from "@elsikora/nestjs-crud-automator";

export const attachmentSource: IApiPolicyAttachmentSource = {
	async getAttachments(principal): Promise<IApiResolvedPolicyAttachments> {
		return {
			attachments: [
				{
					policyId: "allow-managed",
					principalId: principal.id,
					principalType: EApiAuthorizationPrincipalType.USER,
				},
			],
			boundaries: [
				{
					policyId: "boundary-db",
					principalId: principal.id,
					principalType: EApiAuthorizationPrincipalType.USER,
				},
			],
		};
	},
};

export const documentSource: IApiPolicyDocumentSource = {
	async getDocumentsByIds(ids) {
		return ids.map((id) => ({
			document: {
				Statement: [
					{
						Action: ["admin:item:list", "admin:item:read"],
						Condition: {
							StringEquals: {
								"resource.operatorId": "operator-1",
							},
						},
						Effect: "Allow",
						Resource: ["gameport:admin:item/{id}"],
						Sid: id,
					},
				],
				Version: "2012-10-17",
			},
			id,
			namespace: "admin:item",
			sourceType: EApiPolicySourceType.MANAGED,
			version: "2026-03-14",
		}));
	},
};
```

Relation-backed create and update rules can also work directly with raw UUID payloads. If the consumer sends `request.body.operator = "operator-1"` instead of `{ id: "operator-1" }`, IAM conditions can use that value without any hooks fallback:

```typescript filename="authorization/operator-managed-document.ts" copy
const document = {
	Statement: [
		{
			Action: ["admin:item:create", "admin:item:update"],
			Condition: {
				StringEquals: {
					"request.body.operator": "${principal.attributes.operatorId}",
				},
			},
			Effect: "Allow",
			Resource: ["gameport:admin:item/*"],
		},
		{
			Action: ["admin:item:read", "admin:item:list"],
			Condition: {
				StringEquals: {
					"resource.id": "${request.parameters.id}",
					"resource.operatorId": "${principal.attributes.operatorId}",
				},
			},
			Effect: "Allow",
			Resource: ["gameport:admin:item/*"],
		},
	],
	Version: "2012-10-17",
};
```

IAM behavior is intentionally strict:

- explicit deny wins over allow
- boundaries intersect the final allow scope
- the query planner only compiles supported `StringEquals` and `StringEqualsIfExists` conditions on `resource.*` fields marked as `isFilterable`
- controller filters and authorization scopes are merged with logical `AND`, not overwrite semantics
- impossible conflicts collapse to a match-nothing branch instead of rewriting the requested filter
- unsupported or unsafe query-planning cases result in denial instead of hidden in-memory filtering
- duplicate document IDs across sources and missing requested documents throw immediately

## Decision Trace, Audit, and Simulator

Both modes produce the same `IApiAuthorizationDecision` shape:

- `decision.principal`
- `decision.effect`
- `decision.mode`
- `decision.scope`
- `decision.permissions`
- `decision.trace`

Hooks mode populates `decision.trace.rules` and `decision.trace.permissions`. IAM mode populates `decision.trace.attachments`, `decision.trace.boundaries`, `decision.trace.documents`, and `decision.trace.statements`.

Use `ApiAuthorizationSimulator` to evaluate authorization outside HTTP requests:

```typescript filename="authorization/simulator.ts" copy
import { EApiRouteType } from "@elsikora/nestjs-crud-automator";

import {
	ApiAuthorizationSimulator,
	EApiPolicyEffect,
	ErrorException,
} from "@elsikora/nestjs-crud-automator";

const decision = await simulator.evaluate({
	action: EApiRouteType.GET_LIST,
	authorization: itemAuthorization,
	entity: ItemEntity,
	principal,
	requestMetadata: {
		query: {
			limit: 20,
			page: 1,
		},
	},
});

if (decision.effect !== EApiPolicyEffect.ALLOW) {
	throw ErrorException("Access denied");
}
```

If you need audit logging, provide `AUTHORIZATION_AUDIT_SINK_TOKEN` with an `IApiAuthorizationAuditSink` implementation. The runtime records every final decision through that sink.

The built-in permission, attachment, and document resolvers use simple `Map` caches. Call `clear()` on the relevant resolver when your external permission or policy data changes and you need immediate re-evaluation.

## Bootstrap Validation

The authorization module validates configuration during bootstrap. Common failures include:

- securable controllers without an `authorization` block
- controllers with an `authorization` block but no `@ApiControllerSecurable()`
- hooks mode enabled for an entity with no discovered `@ApiAuthorizationPolicy(...)`
- IAM mode without `policyNamespace`, `resourceDefinition`, or registered `documentSources`
- `resourceDefinition.namespace` not matching `policyNamespace`
- `resourceDefinition.entity` not matching the controller entity
- invalid `resourceDefinition.fields` entries or duplicate `path` / `queryPath` mappings

This keeps misconfiguration explicit and prevents silent fallback behavior.

## Migration Notes

When moving from the older authorization API:

1. Replace `subject` with `principal`.
2. Move granted permissions out of the principal and into `hookPermissionSources`.
3. Keep `@ApiControllerSecurable()`, but move mode selection into `@ApiController({ authorization })`.
4. Use `@ApiAuthorizationPolicy(...)` only for hooks mode.
5. Use IAM attachment/document sources for declarative IAM evaluation.

See the [API Reference](/docs/nestjs-crud-automator/api-reference/decorators#apiauthorizationpolicy) for the hooks policy contract and the [controller decorator docs](/docs/nestjs-crud-automator/api-reference/decorators/api-controller/api-controller-securable) for marker behavior.
